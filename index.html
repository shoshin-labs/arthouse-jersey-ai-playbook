For the directors

Amplifying our impact with targeted AI agents.

ArtHouse Jersey does not need a chatbot project. It needs a way to cut administrative friction, protect the core mission, and give staff more time for artists, audiences, and supporters. An AI agent is software with a job: it can work from a shared knowledge base, do one bounded task, check its own work, and hand the result back to a person.

The point

A small fleet of focused agents, working behind the scenes, with staff still making the calls.

The challenge

Administrative friction is eating time that should go into the work.

As ArtHouse Jersey grows, so does the invisible burden of admin. Staff spend time cross-referencing spreadsheets, formatting documents, pulling together context, chasing approvals, and turning the same information into multiple versions for different audiences. None of that is the core mission. It is overhead. The opportunity is to reclaim that time without expanding the headcount. We are not proposing an overhaul of how we interact with the public or our artists. Instead, we are proposing a small fleet of focused agents to work behind the scenes.

What an agent is

Software with a bounded role.

An agent is not an open-ended chatbot. It is given a specific job, a specific source of information, and a clear handoff point to a human.

What it can do

Handle narrow tasks reliably.

It can draft, sort, search, summarise, check, or route work. It should not be asked to improvise where judgement matters.

What it shares

One knowledge base, many boundaries.

The agents can draw from the same internal source of truth while still staying in their lane.

Why it is realistic

Several small gains beat one huge build.

A small fleet is cheaper to start, easier to test, and easier to hand over than one large bespoke system.

Shared base

One secure knowledge base can feed many agents.

This is the part that makes the system cheap enough to start and useful enough to keep. The same core content can sit in one secure place: policies, standard copy, programme history, FAQs, approval rules, venue details, and recurring admin notes. An agent reads only what it needs for the job it is doing. It does not need the whole archive, and it should not see more than it needs. That means the same base can support grant reports, event logistics, artist submissions, policy checks, and meeting notes without duplicating the source material everywhere.

Store once

Keep the source material in one controlled place.

That can be a shared document set, a secure drive, a structured database, or a small internal knowledge system. The point is that staff update one source, not six copies. Keep grant templates, event checklists, artist bios, policy notes, and standard copy in the same controlled place so the agents are drawing from the same material.

Control access

Each agent only sees what it needs.

Some agents may read public copy and policy notes. Others may be limited to specific folders or datasets. Sensitive material stays restricted. A development agent might see donor history; an ops agent might only see event logistics; a policy checker might only see the documents it needs to test against.

Reuse across agents

Different agents can draw on the same base.

A development agent, an operations agent, and a comms agent can all use the same facts without creating three separate knowledge stores. The same brief can also be moved to a different agent later if the team changes the setup.

Move it later

The knowledge should not get trapped in one tool.

If the setup changes, the base can be moved to another agent, another provider, or another system without rebuilding everything from scratch. That keeps the knowledge portable if the charity later wants to change tools or use a different setup.

What this looks like

One source of truth, many narrow workers.

The agent asks for the piece it needs, uses it, and returns the result. The staff team keeps the source content in order.

Why it matters

Security, consistency, and portability.

That gives the charity control now and flexibility later, which is what makes the project sensible rather than experimental.

The case for directors

Four things this gives the charity.

01

Eliminate the admin, keep the art.

Agents can take the first pass at repeat work, so staff spend less time on blank pages, routine follow-up, and searching for the same information twice.

02

Protect the voice.

Because the agents work from ArtHouse Jersey’s own knowledge base and hand the result back to a person, the charity keeps control of tone and messaging.

03

Deepen human relationships.

AI will not build trust, curate work, or nurture a community. Removing admin gives the team more time for the judgment and relationships that matter.

04

Show the result.

One shared setup can support several narrow agents. That keeps the first version as a pilot, so directors can see the benefit before anyone scales it up.

Where outside help fits

Use outside help for the first setup, not forever.

If ArtHouse Jersey wants to move quickly, outside help is useful for mapping the work, wiring the first agents, testing them properly, and leaving the team with something they can own. The aim is a simple system, not a permanent dependency.

1

Map the work

Start with the jobs that waste time every week. The good candidates are grant reports, event logistics, artist submissions, FAQs, meeting notes, and policy checks.

  • List the repeat work.
  • Mark the safe starting points.
  • Agree what always needs a person.

2

Set the boundaries

Keep donor data, staff data, and sensitive documents away from agents that do not need them. Make the rules plain before anything goes live.

  • Define what each agent can see.
  • Define what each agent can send.
  • Define who signs off.

3

Hand it over properly

The first version should come with templates, examples, and a clear owner. If staff cannot run it after handover, the project has not finished.

  • Document the brief.
  • Keep the source of truth tidy.
  • Show the team how to use it without support.

Good outside help

Useful at the start, quieter after launch.

Mapping, integration, testing, and handover are where a small amount of help can save a lot of time.

Not the goal

A project that only works when someone external is in the room.

The system should be simple enough for ArtHouse Jersey to own.

Next steps

Start with one or two low-risk, high-friction jobs.

The first version should be small enough to trust. Once it works, the fleet can grow one agent at a time.

1. Pick the first job

Choose a task people already repeat every week. Good starting points are event copy, donor follow-up, FAQs, grant reports, or meeting notes.

2. Write the brief

State the job, the inputs, the output, and the handoff point. Keep it short and specific.

3. Build the knowledge base

Gather the policies, standard copy, programme history, and FAQs the agent will need. One shared source keeps the answers consistent.

4. Prove it before you grow it

Do not build a big system all at once. Start with one agent, show the benefit, then add the next narrow job.

Example pilot

Take one monthly grant report and turn it into a repeatable workflow.

The agent pulls the latest notes, the relevant programme history, and the approved tone, then produces a first draft for review.

What staff still do

Check the facts, adjust the voice, and sign it off.

The human keeps judgement, relationships, and final approval. The agent only does the repetitive first pass.

What success looks like

Faster first drafts, fewer repeat questions, less rummaging through folders.

If the team can use it without outside support, the pilot has done its job.

Guardrails

The rules that keep the system useful.

Human approval for public voice

Anything that changes the organisation’s tone, promise, or public position gets reviewed by a person before it goes out.

Sensitive data stays controlled

Donor details, staff data, safeguarding matters, and restricted documents should only be exposed to agents that are explicitly approved for that purpose.

Every agent has an owner

Each agent needs one named staff owner, a clear job, and a stop button. No orphaned automation.

Keep a shared source of truth

When an answer matters, the agent should point to the policy, document, or page behind it.

Measure the benefit

Track time saved, fewer repeat questions, faster first drafts, and staff confidence. That is the evidence that the tools are working.

Keep it practical

If a use case does not save time or improve control, leave it alone.

Bottom line

This is not about replacing people. It is about giving them better tools.

A few well-defined agents can take the repetitive work off the team’s plate, keep the knowledge in one place, and leave staff to do the work that needs judgement, taste, and relationships.
